# Implementation Blueprint: Phase 4 - The Lens (Wireless Bridge)

**Date:** 2026-03-17
**Status:** Completed
**Reference:** `docs/technical_plan_2026-03-10.md`

## 1. Objective
Implement the "Lens" system to allow users to capture coin images with their mobile device and transfer them directly to the Patina desktop application via the local network. This eliminates the friction of email/cloud transfer steps.

### Philosophical Alignment
- [x] **Curator-First:** The mobile UI will be a "silent frame"—just a camera viewfinder and a shutter button.
- [x] **Privacy First:** The entire data transfer happens over the local Wi-Fi. No images ever touch the cloud.
- [x] **Single-Click Rule:** The workflow is: Scan QR -> Snap Photo -> Done. The desktop updates instantly.
- [x] **Technical Foundation:** We will use a dedicated Express server within the Main process, strictly isolated from the Renderer's direct control.

---

## 2. Technical Architecture

### A. The Server (Main Process)
- **Technology:** Express.js running on a random available port (or user-configurable range).
- **Location:** `src/main/server.ts`
- **Security:**
  - Binds to local network interface (e.g., `192.168.1.x`).
  - **The Filter:** `multer` middleware validates file type (`image/*`) and size (max 10MB).
  - **Auth:** The QR code URL will include a session-specific token (UUID) to prevent unauthorized uploads if the port is scanned.
- **Routes:**
  - `GET /lens/:token`: Serves the minimal mobile web app (HTML/CSS inlined for simplicity).
  - `POST /lens/:token/upload`: Receives the image, validates it, saves it to `data/images/`, and emits an IPC event to the Renderer.

### B. The Client (Renderer Process)
- **Component:** `LensModal.tsx` (or similar) displaying the QR Code.
- **Library:** `qrcode.react` (or similar lightweight library).
- **Hook:** `useLens()` to manage the server lifecycle (start/stop) and listen for incoming upload events.

### C. Data Flow
1. User opens "Add Coin" -> "Use Lens".
2. Renderer requests `server:start`.
3. Main starts Express, finds local IP, generates a session token.
4. Main returns URL: `http://192.168.1.5:4500/lens/7f8a...`
5. Renderer generates QR code from URL.
6. User scans QR with mobile -> Opens mobile web app.
7. User uploads image.
8. Main validates, saves to `data/images/temp/`, and notifies Renderer via `lens:image-received`.
9. Renderer displays the new image in the "Add Coin" form.

## 5. Security Assessment (`securing-electron`)
**Status:** Verified
### Audit Findings:
- **The Filter:** The blueprint correctly specifies `multer` for file type (`image/*`) and size (< 10MB) validation.
- **Isolation:** The server binds to the local network interface, not `0.0.0.0` globally if possible, or uses a strict firewall rule if not.
- **Auth:** The UUID token in the URL is a critical pre-shared key.
- **CSP:** The mobile page must include a strict Content Security Policy (`default-src 'self'`) to prevent XSS.

### Recommendations:
- **Rate Limiting:** Implement `express-rate-limit` on the upload endpoint to prevent DoS.
- **Sanitization:** Ensure uploaded filenames are sanitized before saving to `data/images/temp/`.
- **Validation:** Do not trust `Content-Type` header alone; check magic numbers if possible (or rely on `multer`'s robust handling).

## 9. Post-Implementation Notes (2026-03-19)

### Issues Encountered and Resolutions

#### Issue 1: File Input onChange Not Firing on Mobile
**Problem:** The `<input type="file" onchange="uploadFile(this)">` was not triggering the upload function on mobile browsers.

**Attempted Solutions:**
1. Used `display: none` on the input - did not work (browsers block hidden file inputs)
2. Used `opacity: 0` with positioning - did not work
3. Used clickable div/button to trigger input click - did not work reliably

**Final Solution:** Use BOTH inline `onchange` AND attach a `change` event listener via `DOMContentLoaded`:
```typescript
document.addEventListener('DOMContentLoaded', () => {
  const fileInput = document.querySelector('input[type="file"]') as HTMLInputElement;
  if (fileInput) {
    fileInput.addEventListener('change', () => {
      uploadFile(fileInput);
    });
  }
});
```

#### Issue 2: Image Path Not Displaying in Desktop App
**Problem:** After successful upload, the image showed as broken in the Edit Coin page.

**Root Cause:** 
- Images were being saved to `{userData}/images/coins/` (Electron's app data directory)
- But `patina-img://` protocol was looking in `data/images/` (project directory)
- Also, `net.fetch('file://...')` was failing

**Fixes Applied:**
1. **Server (server.ts):** Save images to correct location based on environment:
   - Dev: `process.cwd()/data/images/coins/`
   - Prod: `{userData}/images/coins/`

2. **Path Handling (server.ts):** Return relative path for `patina-img://`:
   ```typescript
   const baseDir = isDev 
     ? path.join(process.cwd(), 'data', 'images')
     : app.getPath('userData');
   const relativePath = path.relative(baseDir, req.file.path);
   config.onUpload(relativePath); // e.g., "coins/lens-xxx.jpg"
   ```

3. **Protocol Handler (index.ts):** Use `fs.readFileSync` instead of `file://` URL:
   ```typescript
   const fileBuffer = fs.readFileSync(fullPath);
   return new Response(fileBuffer, {
     headers: { 'Content-Type': `image/${mimeType}` }
   });
   ```

#### Issue 3: Debugging Mobile Without Console Access
**Solution:** Added visible status messages in the mobile HTML:
- Show "Uploading..." before fetch
- Show error messages on failure
- Hide status element when not in use (`display: none`)
